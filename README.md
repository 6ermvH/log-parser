[![Go Report Card](https://goreportcard.com/badge/github.com/6ermvH/log-parser)](https://goreportcard.com/report/github.com/6ermvH/log-parser)

Тестовое задание на стажировку в YADRO, команда прикладной разработки.

Микросервис на Go. Берёт архив диагностики `ibdiagnet2`, разбирает секционный CSV, складывает узлы и порты InfiniBand в PostgreSQL и отдаёт всё это через REST API.

## Что внутри интересного

- Парсинг асинхронный. `POST /parse` отвечает 202 и `log_id`, разбор уходит в горутину. Если приложение умрёт посреди работы, отдельная горутина-reaper подберёт зависшие `processing` и переведёт их в `failed` по таймауту. Подробнее в [Допущениях](#допущения-и-ограничения).
- Тривиальные ошибки ловятся синхронно. Если файла нет или это не zip, клиент получает 400 ещё до того, как в БД появится запись. Битые CSV внутри валидного zip это уже другой случай и асинхронный `failed`.
- Парсер разрезан на два слоя. Стейт-машина знает только про секционный CSV, агрегатор знает только про InfiniBand. Захочется завтра разбирать JSON-lines или бинарь, перепишется один слой. См. [Парсер](#парсер).
- Тестов три уровня: unit, integration на настоящем Postgres через testcontainers, e2e через `httptest` поверх живого роутера. На логических пакетах держим покрытие 77 до 94 процентов. См. [Тестирование](#тестирование).
- Postman-коллекция не пишется руками. Источник правды это аннотации `swag` в коде, из них генерируется OpenAPI 3.1, из неё Postman v2.1. Команда `make docs` собирает обе.

## Запуск

```bash
cp .env.example .env
docker compose up --build
```

Поднимется на `http://localhost:8080`, миграции при первом старте накатятся сами.

Кидаете архив в `data/` (она примонтирована в контейнер как `/app/data`) и отправляете на разбор:

```bash
curl -X POST http://localhost:8080/api/v1/parse \
  -H 'Content-Type: application/json' \
  -d '{"path":"log.zip"}'
```

Ответ 202 и `log_id`, дальше поллите `GET /api/v1/log/{log_id}` пока статус не станет `ok` или `failed`. Если файла нет или это не zip, придёт 400 сразу, ничего в БД при этом не появится.

## Архитектура

Слои стандартные: transport, business, data. Поверх типового CRUD три отступления, ради которых вообще стоило писать README:

1. Парсинг живёт в фоновой горутине, HTTP не ждёт его завершения. Зависшие `processing` подбирает отдельная горутина reaper.
2. Парсер разрезан на формат и доменный агрегатор. Стейт-машина не знает про InfiniBand, агрегатор не знает про zip и поток.
3. Sentinel-ошибки из доменных пакетов проходят через service насквозь. HTTP-слой через `errors.Is` сам решает, какой код вернуть, без слоя кастомных error-типов посередине.

Раскладка по пакетам:

- `cmd/server` точка входа. На старте накатывает миграции, поднимает HTTP, запускает reaper в горутине, по сигналу делает graceful shutdown.
- `internal/api/v1/http` роутер, хендлеры, middleware, JSON-ответы.
- `internal/service` бизнес-логика: `ParseService` для `POST /parse` и `QueryService` для всех GET-ов.
- `internal/parser` сам разбор: стейт-машина и агрегатор поверх неё.
- `internal/storage/postgres` репозиторий поверх `pgxpool`, доменная модель пишется в одной транзакции.
- `internal/storage/migrate` обёртка над `golang-migrate` с миграциями через `embed.FS`.
- `internal/reaper` фоновая чистка зависших `processing`.
- `internal/domain` доменные типы.
- `internal/config` и `internal/logger` конфиг (ENV плюс YAML) и `slog` в stdout.

Компонентная схема (C4 level 3) лежит в [docs/c3-components.md](docs/c3-components.md).

## API

| Метод  | Путь                        | Что делает                                                                       |
| ------ | --------------------------- | -------------------------------------------------------------------------------- |
| `POST` | `/api/v1/parse`             | ставит архив в очередь на парсинг, возвращает `log_id` (202) или 400, если файла нет или это не zip |
| `GET`  | `/api/v1/log/{log_id}`      | текущий статус (`processing`, `ok`, `failed`), счётчики, текст ошибки            |
| `GET`  | `/api/v1/topology/{log_id}` | узлы и порты лога                                                                |
| `GET`  | `/api/v1/node/{node_id}`    | детали узла плюс расширенные блоки (`switch_info`, `system_info`, `sharp_info`)  |
| `GET`  | `/api/v1/port/{node_id}`    | список портов узла                                                               |
| `GET`  | `/health`                   | liveness, ходит в БД                                                             |

Жизненный цикл `POST /parse`: сначала проверяем путь, существование файла и что он действительно открывается как zip. Не прошло, отвечаем 400 и в БД ничего не уходит. Прошло, заводим в `logs` запись со статусом `processing`, клиенту возвращаем `log_id` и 202, а сам разбор уезжает в горутину. Клиент поллит `GET /log/{log_id}` пока не увидит `ok` или `failed`. Если приложение упало посреди разбора, reaper позже переведёт зависшую запись в `failed` по таймауту (см. [Допущения](#допущения-и-ограничения)).

Sequence-диаграммы по каждому эндпоинту вместе с фоновыми горутинами парсера и reaper лежат в [docs/sequences.md](docs/sequences.md).

Postman-коллекция собирается из OpenAPI и лежит в [docs/postman_collection.json](docs/postman_collection.json). Импортируется одним кликом, переменная `{{baseUrl}}` уже выставлена на `http://localhost:8080`. Сама OpenAPI 3.1 спека рядом: [docs/openapi/openapi.json](docs/openapi/openapi.json).

## База данных

Пять таблиц, корнем стоит `logs`, остальное цепляется через FK с `ON DELETE CASCADE`:

- `logs` снапшот загрузки.
- `failed_logs` подробности ошибок разбора, 1:1 к `logs`.
- `nodes` узлы фабрики (host, switch, router, unknown).
- `ports` порты узлов.
- `nodes_info` расширенные блоки (`switch_info`, `system_info`, `sharp_info`) в JSONB.

Полная ERD с кардинальностями в [docs/erd.md](docs/erd.md).

Схема живёт в `migrations/`, накатывается через `golang-migrate` (с `//go:embed migrations/*.sql`) при старте приложения. Отдельного шага в CI нет, миграции едут вместе с релизом.

## Парсер

Чтобы трогать формат отдельно от доменной модели, разбор внутри пакета разрезан надвое:

- Стейт-машина (`internal/parser/statemachine.go`) ходит по файлу построчно, имея три состояния: `Outside`, `Header`, `Body`. Всё, что она знает, это правила секционного CSV: `START_X`, заголовок, данные, `END_X`. На каждой строке данных дёргает колбэк с именем секции, заголовком и значениями. Про InfiniBand там ни строчки.
- Агрегатор (`internal/parser/aggregator.go`) принимает эти события и собирает из них доменную модель: `Log`, внутри `Node`, к `Node` цепляются `Port` и `NodeInfo`. Про zip и поток он, в свою очередь, ничего не знает.

Парсер строгий, любое нарушение формата валит файл целиком. Что именно проверяется, расписано в [Допущениях](#допущения-и-ограничения).

## Про физическую топологию (LINKS)

В `ibdiagnet2.db_csv` нет секции с кабельными связями между портами, она просто не пишется в этот файл. По документации NVIDIA соединения лежат в соседних артефактах того же `ibdiagnet2`: `net_dump`, `lst` и `ibnetdiscover`. Формат там не CSV, а другой табличный текст со своими правилами.

Поэтому `GET /api/v1/topology` сейчас отдаёт только `nodes` и `ports` с метаданными (`state`, `phy_state`, `link_speed_actv`, `link_width_actv`, `lid`, `port_guid`). На «список железа» этого хватает, на полноценный кабельный граф уже нет.

Чтобы добраться до рёбер, нужно научить сервис читать один из `net_dump`, `lst` или `ibnetdiscover`. Для них придётся написать отдельный разборщик, потому что нынешний CSV-парсер не подойдёт. Дальше завести таблицу `connections (port_a_id, port_b_id)` и подмешивать рёбра в ответ `/topology`. В рамках задания этого не делал, на вход приходит только `db_csv`.

## Конфигурация

Через ENV ходят всё, что меняется между окружениями (DSN, порт, путь к данным). YAML лежит рядом и хранит тюнинг внутренних механизмов вроде reaper-а, который не имеет смысла катать через переменные.

| Источник | Параметр         | Назначение                                                 |
| -------- | ---------------- | ---------------------------------------------------------- |
| ENV      | `DATABASE_URL`   | строка подключения к Postgres, обязательно                 |
| ENV      | `PORT`           | порт HTTP-сервера, по умолчанию `8080`                     |
| ENV      | `LOG_LEVEL`      | `debug`, `info`, `warn` или `error`                        |
| ENV      | `DATA_DIR`       | путь к папке с архивами, по умолчанию `./data`             |
| ENV      | `CONFIG_PATH`    | путь к YAML, по умолчанию `./configs/config.yaml`          |
| YAML     | `reaper.tick`    | период проверки зависших `processing`, по умолчанию `30s`  |
| YAML     | `reaper.timeout` | сколько лог может висеть в `processing`, по умолчанию `5m` |

## Допущения и ограничения

Аутентификации нет. Тестовое задание, наружу выставлять не собираемся. В реальном сервисе тут стояло бы middleware с JWT или API-ключом.

Reaper только в одном инстансе. Фоновая горутина, которая подбирает зависшие `processing`, между копиями приложения не координируется. Для нескольких инстансов нужен был бы `pg_try_advisory_lock` или внешний шедулер.

Почему вообще async, а не sync. Архив `ibdiagnet2` на реальной фабрике весит десятки мегабайт, разбор занимает секунды, иногда десятки секунд. Если держать HTTP-соединение до конца разбора, прокси и балансеры начнут резать запросы по своим таймаутам, и клиент в принципе не сможет понять, успел ли парсер дописать данные. Плюс один тяжёлый запрос на всё время блокирует воркер. С `202 Accepted` и поллингом всё проще: `log_id` отдаётся сразу, статус опрашивается когда удобно, retry это просто повторный POST с тем же файлом. Платим за это состоянием `processing` в БД, и если процесс умрёт посреди работы, запись зависнет. Эту проблему и закрывает reaper.

Ошибки разделены на два канала. Если файл не нашёлся или это вообще не zip, проверка на входе сразу возвращает 400, никаких записей в БД при этом не появляется (типичный кейс с опечаткой в пути, не хочется засорять `logs` мусором). Если же zip открылся и разбор начался, но потом что-то сломалось (битый CSV, незакрытая секция, ошибка записи), это уже асинхронный канал: лог едет в `failed`, в stdout улетает структурная slog-запись (`level=WARN`, `err=...`, `parse_duration_ms=...`), а текст ошибки кладётся в `failed_logs(log_id, error_message)` для последующего разбора.

Reaper нужен ровно на тот случай, когда приложение умерло посреди парсинга или клиент оборвал запрос. Запись в `logs` иначе осталась бы в `processing` навсегда. Reaper раз в `reaper.tick` (по умолчанию `30s`) пробегается по таким и переводит в `failed` всё, что висит дольше `reaper.timeout` (`1m` по умолчанию).

Валидация формата CSV жёсткая, но проверок на самом деле всего две:
1. В строке должно быть столько же полей, сколько колонок в заголовке секции.
2. Каждый `START_X` должен закрываться парным `END_X`, иначе ошибка на EOF.

Вторая проверка на практике почти всегда срабатывает через первую: битый или обрезанный `END_X` распознаётся как однотокеновая строка и падает на column count mismatch. До чистой EOF-ошибки доходит только в редком случае, когда все данные валидные, а файл просто обрывается без `END_X`.

Поля портов сохраняются избирательно. В колонках только самые ходовые признаки: `port_state`, `port_phy_state`, `link_speed_actv`, `link_width_actv`, `lid`, `guid`. Остальные примерно 47 полей секции PORTS уезжают в `raw JSONB`. Так не теряем данные и при этом не плодим колонки на всякий случай: если завтра что-то из JSONB понадобится в типизированном виде, перенос это одна миграция.

`sharp_an_info` не валидируется. Файл с key=value блоками парсится в `map[string]string` и кладётся в `nodes_info.sharp_info`. Никаких проверок значений (`endianness`, флаги SHARP) не делаем, это конфиг и его семантика за рамками задания.

Postman собирается полуавтоматически. Источник правды это аннотации `swag` в коде. `make openapi` собирает OpenAPI 3.1, `make postman` гонит её через openapi-to-postmanv2, `make docs` запускает обе цели. Из инструментов нужен только Node.js (через `npx`), Docker не требуется.

Integration и e2e тесты гоняются через `testcontainers-go`. Поднимается настоящий Postgres в Docker, накатываются миграции, тесты бегают по живой БД. В e2e сверху ещё крутится `httptest` с реальным роутером и реальными зависимостями.

## Структура проекта

```text
cmd/server/                  # entry point
internal/
  api/v1/http/               # HTTP handlers, router, middleware, моки
  service/                   # ParseService + QueryService
  parser/                    # стейт-машина + агрегатор
  storage/postgres/          # репозиторий поверх pgx
  storage/migrate/           # golang-migrate runner
  reaper/                    # фоновая чистка зависших логов
  domain/                    # доменные типы
  config/                    # ENV + YAML
  logger/                    # slog JSON
migrations/                  # SQL миграции (embed)
test_migrations/             # сиды для integration-тестов
configs/                     # YAML конфиг (reaper)
tests/e2e/                   # e2e-тесты
testdata/                    # фикстуры (log.zip, битые логи)
docs/                        # диаграммы + OpenAPI + Postman collection
```
